## Branch `fix/launch-hardening` (this branch)

Four launch-hardening fixes from a full pre-launch code review. The miner is
already in production use (a stranger's 4070 rig + two RTX 5090s on vast.ai),
so these target what breaks at 10-100x scale:

1. **Runtime pool failover** (`src/stratum/client.rs`, `src/main.rs`): the
   README advertised failover but only the initial connect walked the `--pool`
   list; the reconnect loop hammered one dead endpoint forever. The client now
   holds the full ordered list, rotates after 2 consecutive failures on the
   current endpoint, and resets to the primary after a successful connect.
2. **Jitter in reconnect backoff** (`client.rs`): deterministic 1,2,4..30s
   backoff meant every miner reconnected in synchronized waves after a pool
   restart (we restart for deploys). Each sleep now gets a random 0.5..1.5
   factor; backoff resets when rotating endpoints.
3. **Rejected shares visible in logs** (`client.rs`, `loop_stratum.rs`):
   rejects only bumped an atomic that the GUI launcher polled; headless users
   (HiveOS/systemd, the majority) had zero signal while losing revenue. Now a
   `tracing::warn!` per reject with the pool's error payload, plus a running
   accepted/rejected tally in the periodic hashrate line.
4. **catch_unwind around OpenCL auto-probe** (`main.rs`): broken AMD ICDs can
   panic during enumeration; the `auto` path now falls through to CPU like the
   CUDA and forced-opencl paths already did.

State: builds clean (`cargo build --release`), 70/70 tests pass, pushed for
review. Review found further should-fix items not included here: per-SM CUDA
geometry scaling (5090s likely leave hashrate on the table with 560 fixed
blocks), checksum verification in install.ps1, CI `--locked` + toolchain
pinning, Windows code signing.
