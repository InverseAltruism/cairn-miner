# 2026-07-19 (part 2) — mats (with Claude)

## The v0.2.3 duplicate "fix" was WRONG — real fix on `fix/cuda-stale-nonce-guard`

Honest correction: the `fix/gpu-duplicate-shares` fix shipped in **v0.2.3 does
NOT resolve the duplicate shares.** I verified it on a 12-min window that
happened to be quiet (1/62) and released on that — a mistake. A proper look
showed ~7-10% duplicate rejects persisted on the RTX 2080 at low diff.

### Real root cause (proven, not guessed)
The duplicates are a **timing race in the CUDA 2-stream backend**: a device
`found_flag`/`found_nonce` occasionally survives across `hash_range` calls and
is drained as a stale "find" from an earlier, lower chunk. Proven by
instrumenting the find path to log whether the returned nonce is inside the
chunk it was asked to scan — caught `inrange=false` (GPU returned a nonce
OUTSIDE the requested range). The race is so timing-sensitive that even ONE
log line per find masked it — which is why my earlier "fixed binary" and the
debug builds looked clean. The loop_stratum cursor/xn2 sweep is CORRECT
(monotonic, verified) — the bug is purely in the backend.

The in_flight reset (v0.2.3) was correct hygiene (mirrors OpenCL) but did not
close the race.

### The fix
One line in `src/backends/cuda.rs` `hash_range`: discard any drained result
whose nonce is outside `[nonce_start, nonce_end)`. The caller sweeps
monotonically in disjoint chunks, so a stale nonce is ALWAYS < nonce_start →
always caught. It's a pure filter (no I/O), so unlike logging it doesn't
perturb the race timing. Makes hash_range's contract ("only return a nonce I
actually found in range") hold unconditionally.

### Verification (this time: HOURS, on a non-logging binary)
Guard binary (sha 8c2ca01f), single process, RTX 2080 at low diff:
- baseline 17:33 → +2.5h at 20:03: **647 accepted, 0 rejected, 0 duplicates.**
- vs ~7-10% before. Clean across the whole window, not a lucky sample.
